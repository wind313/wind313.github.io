TCP的可靠性

TCP 是通过连接管理、序列号、确认应答、重发机制、流量控制、拥塞控制等机制实现可靠性传输的。

重传机制

常见的重传机制有：超时重传、快速重传、SACK、D-SACK

超时重传

发送数据时，设置一个定时器，超过指定时间没有收到ACK确认应答报文，就会重发数据。

数据包丢失和确认应答报文丢失时都会触发超时重传。

RTT（数据包从发出去到收到ACK的时间）是经常变化的，因为网络是时常变化的，所以RTO（重传超时时间，应该略大于RTT）也应该是动态变化的值。

TCP采用一些算法：加权移动平均（EWMA）算法、Karn算法，Jacobson算法。这几个算法结合起来根据RTT的变化来估计RTO的值。

RTO的计算：

  1. 采样

    • 采样RTT，进行加权平均，算出一个SRTT（平滑RTT），这个值是不断变化的。
    • 采样RTT波动范围，避免RTT大的波动很难被发现。

  2. 第一次计算

    • 使用第一个RTT测量值来初始化SRTT和DevRTT（RTT测量值与SRTT之间的差异）。

  3. 后续计算

    使用新的RTT测量值更新SRTT和SevRTT，更新公式包括平滑因子来调整SRTT和DevRTT，再使用上面公式计算新的RTO。

  • EWMA：用在 SRTT 和 DevRTT 的计算中，用于平滑 RTT 的波动。
  • Karn ：处理重传时的 RTT 测量问题，确保只有在成功接收到 ACK 时才更新 RTT。
  • Jacobson：将这两者结合，并提出了 RTO 的计算公式，确保 RTO 能够动态适应网络状况。

一个数据包每次重传时间间隔为前一次的2倍（即重传间隔指数递增），如果两次超时，说明网络环境差，不建议频繁发送。

快速重传

快速重传不已时间为驱动，而是以数据驱动重传。

  1. 接收方收到1号报文，ACK回2（收到报文的ACK+1）；
  2. 2号报文没有收到，3号报文收到了，ACK回2；
  3. 后面收到报文，ACK继续回2；
  4. 发送方收到三个相同ACK，会在超时之前，重传2号报文；
  5. 接收到收到2号报文，由于3、4、5号报文都收到，ACK回6。

如果2、3号报文都丢失，4、5、6号报文收到，ACK回2，发送方不知道ACK2是哪个报文的ACK（即不知道2之后有没有丢失报文）。为了解决这个问题，有了SACK方法。

SACK方法

SACK（Selective Acknowledgement，选择性确认）需要在ACK的TCP头部加一个SACK字段，将已收到数据的信息发送给发送方，发送方就只带哪些数据收到和哪些没有收到，就可以只重传丢失数据。

发送方和接收方都要支持SACK才能打开SACK，Linux可以通过net.ipv4.tcp_sack参数打开，Linux2.4之后默认打开。

Dupliacte SACK

Duplicate SACK（D-SACK）主要使用SACK告诉发送方哪些数据被重复接收。

优点：

  • 可以让发送方知道是发送的包丢失还是ACK包丢失；
  • 可以知道是不是发送方数据包被网络延迟；
  • 可以知道网络中是不是把发送方的数据包复制。

Linux可以通过net.ipv4.tcp_dsack参数开启，Linux2.4后默认打开。

流量控制

流量控制就是发送方根据接收方的实际接收能力控制发送的数据量，TCP 利用滑动窗口实现流量控制。

滑动窗口

窗口实际上是操作系统开辟的一个缓存空间，发送方在等到ACK返回前，必须在缓冲区保留已发送的数据，只有按期收到ACK，才可以从缓存区清除。窗口大小就是无需等待确认应答可以继续发送数据的最大值。

TCP头里有一个Window（窗口大小）字段，接收方通过这个字段告诉发送端自己还有多少缓存区可以接受数据。

如果接收方资源紧张，操作系统可能直接减少接收缓冲区大小，发送方还不知道接收窗口收缩，发送数据大小超过接收窗口，数据包会丢失，所以TCP规定先收缩窗口，在减少缓存，就可以防止这种情况发生。

发送方滑动窗口

分成四个部分：

  • 已发送并受到ACK的数据；
  • 已发送未收到ACK的数据；
  • 未发送但总大小在接收方处理范围内；
  • 未发送但总大小超过接收方处理范围。

程序使用窗口大小和三个指针（两个绝对指针，一个相对指针）跟踪四个部分：

  • SND.WND：发送窗口的大小（由接收方指定）；
  • SND.UNA(Send Unacknowleged)：绝对指针，指向已发送但未收到ACK（上面第二部分）的第一个字节的序列号；
  • SND.NEXT：绝对指针，指向未发送但可发送范围（上面第三部分）的第一个字节的序列号；
  • 相对指针，指向第四部分的第一个字节，需要SND.YNA指针加上SND.WND大小的偏移量。

可用窗口大小 = SND.WND -（SND.NXT - SND.UNA）。

接收方滑动窗口

分为三个部分：

  • 已接收并确认的数据（等待应用程序读取）；
  • 未收到数据但可以接收的数据；
  • 未收到数据并不可以接受的数据。

使用两个指针划分：

  • RCV.WND：接收窗口大小，回通知给发送发；
  • RCV.NXT：指针，指向期望收到的下一个数据字节的序列号，也就是第二部分的第一字节；
  • 相对指针，指向第三部分的第一个字节，需要RCV.NXT指针加上RCV.WND大小的偏移量。

接收窗口约等于发送窗口大小，不完全相等。因为接收窗口大小不是一成不变的，而且是通过TCP报文告诉发送方的，传输过程存在延时。

窗口关闭

窗口关闭就是窗口变为0时，发送方不会发送数据，直到窗口变为非0为止。

接收方除了收到数据包会响应ack，接收窗口大小发生显著变化也可以发送一个窗口更新的ACK。

如果接收方处理完数据，接收窗口变为非0，就会通过ACK报文通知发送方窗口大小，如果该ACK丢失，就会导致发送方一直等待接收方的非0窗口通知，接收方也一直等待发送方的数据，会造成死锁。

TCP为每个连接设置一个定时器，只要一方收到对方0窗口通知，就启动定时器，如果超时，就会发送Window probe（窗口探测）报文，接收方确认这个报文时，给出自己接收窗口大小。如果接收窗口还是0，重新启动计时器。窗口探测次数一般3次，每次30-60秒，3次后还是0，就会RST报文中断连接。（不同的TCP实现可能不一样）

糊涂窗口综合症

糊涂窗口综合症就是接收方腾出几个字节并告诉发送方现在有几个字节的窗口，发送方义无反顾发送这几个字节。TCP+IP头有40个字节，发送四十几个字节只为了传输几个字节，不划算。

解决方式：

  1. 接收方不通知小窗口：窗口大小小于min（MSS，缓存空间/2）时，向发送方通知窗口为0；
  2. 发送方避免发送小数据：Nagle算法，只有满足以下任一条件，才可以发送数据：
    • 窗口大小>=MSS&&数据大小>=MSS；
    • 收到之前发送数据的ack。

以上两个结合才能避免糊涂窗口综合症，Nagle算法默认是打开的，对于一些小数据包交互的场景（比如telnet、ssh），需要关闭Nagle算法。可以在Socket设置TCP_NODELAY选项关闭该算法（没有全局参数）。

拥塞控制

拥塞控制就是根据当前网络状况调整发送数据速度。

cwnd（拥塞窗口）变化规则：

  • 网络中没有阻塞，cwnd增大；
  • 网络中出现阻塞（没有在规定时间内收到ACK），cwnd减小。

拥塞控制有四个算法：慢启动、拥塞避免、拥塞发生、快速恢复。

慢启动

发送方每收到一个ACK，cwnd就会+1。

  1. 初始化cwnd为1（假设为1，Linux可以使用ss -nli命令查看每个TCP连接的cwnd初始值），开始发送数据包，发包个数为1；
  2. 收到ACK，此时cwnd = cwnd+1=2，发包个数为2；
  3. 收到2个ACK，cwnd = cwnd+2=4，发包个数为4；
  4. 收到4个ACK，cwnd = cwnd+4=8，发包个数为8；

……

慢启动发包个数指数型增长，当cwnd<ssthresh（slow start threshold，慢启动门限，一般为65535字节）时，使用慢启动，否则使用拥塞避免。

拥塞避免

发送方每收到一个ACK，cwnd = cwnd+1/cwnd。

假定ssthresh为8，此时cwnd为8：发送方收到8个ACK，每个+1/8，一共+1，即cwnd=cwnd+1=9。

拥塞避免发包个数是线性增长，cwnd增长过程中，网络会进入阻塞状况，出现丢包，触发重传机制，就会进入拥塞发生。

拥塞发生

超时重传和快速重传的拥塞发送算法是不同的。

  1. 超时重传
    • ssthresh=cwnd/2；
    • cwnd恢复为初始值。
    • 重新开始慢启动。
  2. 快速重传
    • cwnd = cwnd/2；
    • ssthresh = cwnd；
    • 进入快速恢复
快速恢复
  1. cwnd = ssthresh +3（收到3个ACK）；
  2. 重传丢失数据包；
  3. 如果收到重复ACK，cwnd+1；
  4. 如果收到新数据ACK，把cwnd设置为ssthresh值，因为ACK确认了新数据，说明duplicate ACK都收到，恢复过程结束，恢复到之前状态，进入拥塞避免。
